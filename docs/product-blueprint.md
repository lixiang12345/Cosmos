# Cosmos 产品蓝图（工作名）

> 文档状态：原型阶段基线  
> 产品类型：面向软件研发团队的事件驱动 AI Agent 控制台  
> 参考方向：Augment Code 的 Agent/Expert、GitHub、Slack 协作体验，但不做界面逐像素复制  
> 当前目标：先落地可验证产品价值与关键交互的高保真可点击原型，再进入真实集成开发

## 1. 产品定位

### 1.1 一句话定位

Cosmos 是软件研发团队的 AI 工程专家编排与治理平台：团队可以创建可复用的“专家”，让它们由人工、GitHub 或 Slack 事件触发，在隔离环境中完成代码任务，并在关键风险点请求人工审批。

### 1.2 产品不是

- 不是只有代码补全能力的 IDE 插件。
- 不是一次性问答聊天机器人。
- 不是无审批、无审计的全自动生产发布工具。
- 不是通用低代码工作流平台；首版只服务软件工程任务。

### 1.3 核心价值主张

1. **异步执行**：开发者提交目标后可以离开，系统持续推进并在需要决策时召回人。
2. **专家复用**：把团队规范、工具权限、触发条件和处理步骤固化为可复用专家，而不是反复写提示词。
3. **跨工具闭环**：从 Slack/GitHub 发起，经代码修改、检查、PR、Review 反馈到合并建议，过程不丢上下文。
4. **可见、可控、可审计**：每次计划、工具调用、代码产物、审批和外部写操作都有状态、依据与审计记录。

### 1.4 差异化原则

- **从“聊天”升级为“运行系统”**：主要对象是任务、运行、事件、产物和审批，而不只是消息。
- **从“单 Agent”升级为“专家协作”**：一个工作流可由 PR 作者、Reviewer、修复专家等不同职责协同。
- **从“黑盒执行”升级为“风险分级”**：低风险步骤自动推进，高风险步骤必须说明影响并等待授权。
- **从“网页孤岛”升级为“团队工作入口”**：Slack 与 GitHub 既能触发任务，也能接收进度、审批和结果。

## 2. 目标客户与痛点

### 2.1 优先客户

| 优先级 | 客户/角色 | 典型特征 | 购买或使用动机 |
| --- | --- | --- | --- |
| P0 | 20-500 人研发组织的研发效能、平台工程团队 | GitHub + Slack 使用成熟，有内部规范和重复任务 | 提升工程吞吐，统一 Agent 治理与安全边界 |
| P0 | Engineering Manager / Tech Lead | PR 积压、跨仓库协作多、Review 资源不足 | 缩短交付周期，减少人工追踪和上下文切换 |
| P0 | 日常开发者 | 同时处理需求、修复、Review、文档和发布琐事 | 把明确但耗时的任务交给 Agent 异步完成 |
| P1 | DevOps / SRE / 安全工程团队 | 自动化脚本多但分散，变更风险高 | 将诊断、修复建议、变更审批纳入同一审计链路 |
| P1 | 多项目软件交付团队 | 仓库和客户规范多，人员切换频繁 | 复用项目上下文和标准化执行流程 |

### 2.2 核心痛点与产品响应

| 客户痛点 | 当前代价 | Cosmos 的产品响应 |
| --- | --- | --- |
| AI 每次都要重新理解仓库、规范和工具 | 提示词重复、结果不稳定、个人经验无法沉淀 | 专家模板保存指令、知识源、工具、权限和验收规则 |
| 工程任务分散在 Slack、Issue、PR 和本地终端 | 上下文切换频繁，任务容易失联 | 外部事件触发统一 Session，并把状态与结果回写原线程 |
| Agent 执行过程不透明 | 团队不敢授权高价值任务，失败后难定位 | 时间线展示计划、步骤、工具调用、差异、检查与决策依据 |
| 自动化脚本只覆盖固定路径，遇到异常就中断 | 维护成本高，仍需人工拼接后续动作 | Agent 在约束内动态规划，异常时生成可操作的恢复建议 |
| PR 创建、Review、修改意见往返耗时 | Lead 成为瓶颈，交付周期拉长 | PR Author、Reviewer、Review Fixer 专家组成可审批闭环 |
| 权限、密钥和生产变更风险难治理 | 安全团队阻止规模化采用 | 最小权限、环境隔离、审批门、秘密引用和完整审计日志 |

### 2.3 关键待办任务（Jobs to Be Done）

- 当我在 Slack 描述一个明确的代码任务时，我希望系统自动找到正确仓库并创建 PR，让我只在需要判断风险时介入。
- 当 GitHub PR 进入 Review 状态时，我希望 Reviewer 专家自动检查团队规则并给出有证据的意见。
- 当 Reviewer 留下修改意见时，我希望原任务上下文继续被使用，而不是让开发者重新解释全部背景。
- 当 Agent 准备执行有风险的写操作时，我希望看到影响范围、检查结果和回滚方式，再决定是否批准。
- 当自动化失败时，我希望能从日志、工具调用和产物快速判断失败原因，并重试失败步骤。

## 3. MVP 范围

### 3.1 MVP 必须具备

| 能力域 | MVP 能力 | 原型表现 |
| --- | --- | --- |
| 组织与仓库 | 单组织、多仓库；GitHub App 授权与仓库选择 | 可完成连接向导，展示已连接、权限不足和断开状态 |
| 专家 | 创建、编辑、启停专家；配置职责、指令、仓库、环境、工具与审批策略 | 完整表单、校验、预览、保存和未保存提示 |
| 会话 | 人工创建任务；查看 Session 列表和详情；暂停、取消、重试 | 使用 Mock 数据演示完整状态变化 |
| 执行可见性 | 展示计划、步骤、工具调用摘要、日志、文件差异、测试和产物 | 时间线可折叠，默认突出结论与异常，不堆满原始日志 |
| 触发器 | GitHub PR 事件、Slack Mention、手动触发 | 可在专家内配置，并在触发器页测试示例事件 |
| 审批 | 高风险动作进入审批；批准、要求修改、拒绝 | 有统一审批收件箱，并可在 Session 内就地处理 |
| 外部回写 | Slack 线程回写状态/审批/结果；GitHub 创建 PR 和评论 | 原型展示消息预览及深链；真实 MVP 使用正式 API |
| 审计 | 外部事件、用户操作、Agent 写操作、审批和结果可检索 | 事件日志支持类型、来源、状态与时间筛选 |
| 安全 | Secret 只保存引用；权限最小化；敏感值脱敏 | UI 不回显 Secret；危险授权有明确风险提示 |

### 3.2 MVP 专家模板目录

原型采用 [Augment Workflows](https://www.augmentcode.com/workflows) 的公开模板目录作为功能范围参考。去重后共 62 个模板，Cosmos 保留工作流意图与分类，但使用自己的品牌、界面、权限模型和实现，不复制对方代码或视觉资产。

| 分类 | 模板数 | 代表模板 |
| --- | ---: | --- |
| Coding | 19 | Ticket or task to a merged PR、Automated deep review on every PR、Figma design to production code |
| DevOps | 8 | CI build failure auto-fix、Safe dependency update with regression loop |
| QA | 7 | E2E Playwright verification on every PR、Performance regression gate |
| Security | 6 | Threat model from PRD or RFC、OWASP vulnerability scanner |
| Support | 6 | Intelligent ticket dispatcher、Doc gap auto-fix from support tickets |
| Observability | 5 | Alert to root-cause analysis in Slack、Runbook auto-execution on incidents |
| Data | 3 | AI-assisted backlog prioritization、Weekly product digest |
| Release | 3 | Release candidate readiness review、Multi-audience release notes |
| Database | 2 | Safe schema migration pipeline、Heavy query sandbox optimizer |
| AI / ML | 2 | Synthetic evaluation dataset generation、Prompt eval regression gate |
| Network | 1 | Network health and intrusion watch |

全部模板以稳定 `id`、分类、名称、说明和来源路径存放在 `src/data/expertTemplates.ts`，专家库和新建任务选择器共用同一数据源。首个演示闭环使用 Ticket or task to a merged PR、Automated deep review on every PR 和 Unit-test coverage gap analysis and draft PR。

### 3.3 首版明确不做

- 不做模型训练、微调或自建模型托管。
- 不做完整 IDE、代码补全或本地编辑器替代品。
- 不做开放式第三方插件市场；仅提供受控的内置工具与 MCP 接入占位。
- 不做任意生产基础设施写操作和自动发布；生产级动作只展示审批设计。
- 不做复杂多租户计费、用量套餐和财务结算。
- 不做企业级 SAML/SCIM、数据驻留、多区域部署；架构需预留但不阻塞 MVP。
- 不做可视化拖拽工作流编排器；首版通过专家 + 触发器 + 策略表达流程。
- 不做完整移动端控制台；小屏仅保证查看状态和处理审批，重配置仍在桌面端完成。

## 4. 信息架构

### 4.1 核心术语

| 术语 | 定义 | 界面命名原则 |
| --- | --- | --- |
| Expert / 专家 | 可复用的 Agent 定义，包含职责、上下文、工具、环境和策略 | 中文界面统一用“专家”，不混用机器人、助手 |
| Session / 会话 | 一次目标驱动的持续执行实例，可由人或事件创建 | 列表和详情统一用“会话” |
| Run / 运行 | 会话中的一次执行尝试；重试会产生新 Run | 面向普通用户可显示为“第 N 次运行” |
| Step / 步骤 | 运行中的计划或工具执行单元 | 以状态和结果为主，不暴露内部推理 |
| Trigger / 触发器 | 启动专家的手动或外部事件条件 | 不使用含混的“订阅”作为主导航名称 |
| Environment / 环境 | Agent 执行代码和工具的隔离运行空间 | 明确仓库、分支、资源与网络策略 |
| Approval / 审批 | 对高风险写操作的人工决策记录 | 显示风险、影响、证据和过期时间 |
| Artifact / 产物 | Diff、提交、PR、报告、测试结果等可交付结果 | 在会话和文件视图中可追踪 |

### 4.2 全局导航

```text
Cosmos
├── 新建会话
├── 会话
├── 审批
├── 代码库
├── 配置
│   ├── 专家
│   ├── 环境
│   ├── 触发器
│   ├── 集成
│   ├── MCP 注册表（MVP 占位/只读）
│   ├── Webhooks
│   ├── 事件日志
│   └── Secrets
└── 账户与组织
    ├── 成员与角色（MVP 只读）
    └── 设置
```

### 4.3 会话详情的局部架构

```text
会话标题 / 状态 / 来源 / 更多操作
├── Agent：目标、时间线、计划、工具结果、审批、最终总结
├── Terminal：当前运行的终端输出与命令历史
├── Files：变更文件、Diff、产物
└── Triggers：本会话来源、后续事件监听与关联外部线程
```

其中 Agent 是默认视图。Terminal 和 Files 是证据视图，不应抢占主任务叙事；Triggers 在原型中可使用 “触发” 命名，避免与计费订阅混淆。

## 5. 关键业务闭环

### 5.1 人工发起到 PR

1. 用户点击“新建会话”，选择专家与代码库，描述目标和验收条件。
2. 系统校验专家权限、环境和仓库连接后创建 Session。
3. Agent 生成可执行计划，开始读取代码、修改文件和运行检查。
4. 时间线持续更新；低价值工具细节默认折叠，异常和决策点展开。
5. 创建 PR 前显示变更摘要、测试结果和风险级别。
6. 策略允许时自动创建 Draft PR；否则进入人工审批。
7. 最终结果包含 PR 链接、变更、验证、已知风险和建议下一步。

### 5.2 Slack 触发到人工审批

1. 用户在授权频道 Mention Cosmos 专家并描述任务。
2. Slack App 立即确认已接收，展示专家、仓库和“查看会话”深链。
3. Agent 在后台运行，过程更新集中在原线程，避免刷屏。
4. 需要审批时，线程消息展示风险原因、影响文件、测试状态和操作选项。
5. 用户在 Slack 或 Web 控制台批准、要求修改或拒绝；同一审批只接受第一次有效决策。
6. Agent 继续运行并回写 PR/结果；所有外部消息和审批同步到审计日志。

### 5.3 GitHub PR Review 闭环

1. GitHub `pull_request.ready_for_review` 事件匹配 Reviewer 专家触发器。
2. 系统去重事件并创建/关联 Session，锁定 PR 当前 Head SHA。
3. Reviewer 分析 Diff、相关代码、测试结果和仓库规范。
4. 低置信度或高风险问题以 Review 评论提出；结论必须带文件或检查证据。
5. Review Fixer 监听评论，处理可执行修改并逐条回复。
6. 当 Head SHA 变化时重新验证，避免在过期代码上给出批准。
7. 满足策略后提示“可合并”；合并本身默认保留人工审批。

### 5.4 失败与恢复闭环

1. 步骤失败后标明失败类型：权限、环境、工具、测试、冲突、超时或未知。
2. 可恢复错误按退避策略自动重试；不可恢复错误进入“需要输入”。
3. 用户可以补充信息、修改参数、重试失败步骤或从新分支重新运行。
4. 新 Run 保留旧 Run 的日志和产物引用，界面明确标识当前有效版本。

## 6. 页面与交互规范

### 6.1 应用框架（App Shell）

- 桌面端采用左侧固定导航、顶部会话栏、右侧主工作区的安静型工作台布局。
- 1440px 宽度下侧栏建议 240px；可折叠到 56px 图标栏。顶部栏建议 48px。
- 主内容不使用悬浮大卡片堆叠；时间线、工具栏和详情区以分隔线、层级底色组织。
- 用户收起侧栏后保持状态；小于 1024px 时侧栏改为抽屉。
- 顶部持续显示当前组织、会话状态和全局运行指示，不因切换 Tab 丢失。

### 6.2 新建会话 `/sessions/new`

**必要字段**

- 专家：可搜索选择，默认使用最近一次。
- 代码库：只展示该专家有权限的仓库。
- 基础分支：默认仓库默认分支，可修改。
- 任务描述：支持多行文本、粘贴链接和附件。
- 验收条件：可选，但提供结构化建议项。

**关键交互**

- 选定专家后即时展示其工具、环境、审批策略和最近成功率摘要。
- 输入 GitHub/Slack 链接时解析为可移除的上下文项，失败时保留原文本。
- 创建前做权限与连接检查；不满足时给出明确修复入口。
- 主操作为“开始运行”，次操作为“保存草稿”。禁止使用无含义的“提交”。

### 6.3 会话列表 `/sessions`

**信息密度**

- 默认以紧凑列表展示标题、专家、仓库、来源、状态、更新时间、PR/审批标识。
- 支持按状态、专家、仓库、来源和时间筛选；支持标题与 PR 编号搜索。
- “运行中”“等待审批”“失败”在视觉上优先于普通完成记录。

**状态与操作**

- 空状态：说明当前没有会话，并提供“新建会话”和“连接集成”两个相关入口。
- 加载状态：保持行高稳定，使用骨架占位，不显示整页旋转器。
- 批量操作首版只支持归档；取消运行必须逐项确认。
- 单会话支持收藏、重命名、归档、恢复和删除；菜单支持点击外部与 `Esc` 关闭，危险删除使用二次确认。
- 原型阶段将会话管理状态写入 `localStorage`，用于验证刷新后的连续性；真实 MVP 以服务端 Session 状态为准。

### 6.4 会话详情 `/sessions/:sessionId`

**时间线结构**

1. 用户目标或外部事件卡片。
2. Agent 计划及计划调整说明。
3. 步骤组：读取、修改、测试、外部调用。
4. 审批卡片或“需要输入”卡片。
5. 最终结果与产物。

**展示规则**

- 不展示模型私有推理过程；展示面向任务的计划、已执行动作、依据和结果。
- 连续同类工具调用合并为一组，例如“读取 8 个文件”；用户可展开查看明细。
- 命令默认显示命令摘要、退出码和尾部关键日志，完整日志放入 Terminal。
- 变更结果显示文件数、增删行、测试状态和 PR 链接。
- 状态变化通过文本 + 图标 + 色彩共同表达，不只依赖颜色。
- 新事件到达时，如果用户停留在底部则自动跟随；用户向上阅读时显示“3 条新动态”按钮，不强制跳动。

**输入区**

- 固定在内容底部，包含附件、模型/执行策略选择、发送和停止按钮。
- 运行中发送的新消息进入队列，并明确标注“下一步执行”；用户可撤回尚未消费的消息。
- 等待审批时输入区仍可补充信息，但审批主操作固定在审批卡片上。
- 熟悉图标按钮必须有 Tooltip 和可读的 `aria-label`。

### 6.5 审批收件箱 `/approvals`

- 默认按“等待我处理”排序，展示风险等级、动作、会话、仓库、发起时间和过期时间。
- 详情必须包含：Agent 想做什么、为什么需要审批、影响范围、验证结果、回滚/恢复方式。
- 决策包含“批准”“要求修改”“拒绝”；拒绝和要求修改必须填写原因。
- 批准危险操作前进行二次确认，但不使用输入固定短语等高摩擦设计。
- 审批过期、已被他人处理或关联 SHA 变化时，按钮失效并说明原因。

### 6.6 专家列表与编辑器 `/configuration/experts`

**列表**

- 显示专家名称、职责、状态、绑定仓库、触发器数、最近运行和成功率。
- 提供从模板创建、复制、启停、归档；删除只允许对无历史引用的草稿专家使用。

**编辑器分区**

1. 基本信息：名称、描述、头像色/图标。
2. 指令与验收：职责、约束、完成定义。
3. 上下文：仓库、路径范围、知识文件。
4. 工具与权限：只读、写代码、建 PR、评论等细粒度开关。
5. 环境：运行镜像、资源、超时、网络策略。
6. 触发器：手动、Slack、GitHub 事件及条件。
7. 审批策略：按动作和风险等级配置。
8. 测试：使用示例事件运行 Dry Run，展示将执行的步骤但不产生外部写操作。

采用左侧分区导航 + 单层表单，不把每个分区包进嵌套卡片。离开未保存页面时必须提醒。

### 6.7 集成 `/configuration/integrations`

- 首版展示 GitHub 与 Slack；每项包含连接状态、授权范围、安装账户、最近事件和诊断入口。
- 连接流程采用分步向导：选择账户 -> 授权范围 -> 选择资源 -> 测试 -> 完成。
- 权限不足要说明缺失权限以及为什么需要，不能只显示“连接失败”。
- 断开集成前列出受影响专家和触发器，并要求确认。

### 6.8 事件日志 `/configuration/events`

- 以表格展示时间、来源、事件、关联对象、处理状态、重试次数和耗时。
- 事件详情使用右侧抽屉，包含脱敏 Payload、匹配规则、处理步骤和错误。
- 支持复制事件 ID、重放可重放事件；重放前显示幂等性和外部写入风险。

### 6.9 Slack 交互规范

- 不在 Web 产品内复制 Slack UI，只在集成设置和 Demo 场景中提供消息预览。
- 首次响应应在数秒内确认任务已收到；长任务只在关键阶段更新。
- 每条状态消息同时包含纯文本结论和“查看会话”深链。
- 审批消息突出风险与变更，不把完整日志塞入线程。
- 最终消息结构固定为：结果、验证、PR/产物、风险或需要用户处理的事项。

## 7. 状态机与领域模型

### 7.1 Session 状态机

```text
draft -> queued -> running -> succeeded
                    |  |  \
                    |  |   -> failed -> retrying -> running
                    |  -> waiting_input -> running
                    -> waiting_approval -> running

queued/running/waiting_* -> cancelling -> cancelled
succeeded/failed/cancelled -> archived
```

规则：

- 一个 Session 可包含多个 Run，但同一时间只有一个活动 Run。
- `waiting_input` 与 `waiting_approval` 必须区分，前者需要补充上下文，后者需要授权动作。
- 所有终态必须有完成摘要或结构化失败原因。

### 7.2 Approval 状态机

```text
pending -> approved
        -> changes_requested
        -> rejected
        -> expired
        -> invalidated
```

`invalidated` 用于代码 SHA、参数或权限在审批后发生变化的情况。审批决策必须幂等并记录决策人、渠道、时间和理由。

### 7.3 核心领域对象

| 对象 | 关键字段 | 主要关系 |
| --- | --- | --- |
| Organization | id, name, policySetId | 拥有成员、仓库、集成、专家 |
| User | id, identity, role, status | 发起会话、处理审批 |
| Repository | id, provider, owner, name, defaultBranch, installationId | 属于组织，绑定专家与会话 |
| Integration | id, type, status, scopes, externalAccount | 接收事件、执行外部动作 |
| Environment | id, image, resourceLimit, timeout, networkPolicy | 被专家选择，被 Run 实例化 |
| Expert | id, name, instructions, toolPolicy, approvalPolicy, version, status | 绑定仓库、环境、触发器 |
| ExpertVersion | id, expertId, configSnapshot, createdBy | Session 固定引用某个版本，保证可复现 |
| Trigger | id, source, eventType, filter, expertId, enabled | 外部事件命中后创建 Session |
| Session | id, title, source, expertVersionId, repositoryId, state | 包含消息、Run、产物、审批 |
| Run | id, sessionId, attempt, headSha, state, startedAt, endedAt | 包含 Plan、Step 和 ToolCall |
| Step | id, runId, type, title, state, summary | 关联工具调用与产物 |
| ToolCall | id, stepId, tool, inputRef, outputRef, exitCode | 产生审计事件，敏感字段脱敏 |
| Artifact | id, sessionId, type, uri, sha, metadata | 表示 Diff、Commit、PR、报告、测试 |
| Approval | id, sessionId, action, risk, state, evidence, expiresAt | 由用户决策并控制 Run |
| ExternalEvent | id, integrationId, externalId, type, payloadRef, receivedAt | 经过幂等去重后匹配 Trigger |
| AuditEvent | id, actor, action, target, channel, result, timestamp | 记录所有重要读取、写入和决策 |

### 7.4 领域约束

- Session 必须固定 `ExpertVersion`，专家后续编辑不能静默改变已运行会话。
- ExternalEvent 以服务商事件 ID + 安装范围做幂等键。
- 外部写操作必须关联 Session、Run、ToolCall 和操作者（用户或专家）。
- Secret 只保存 Vault/Provider 引用，任何日志与 Mock 数据都不得包含明文值。
- PR 相关审批必须绑定 Head SHA；SHA 变化后原批准自动失效。

## 8. 视觉系统

### 8.1 设计气质

关键词：专业、克制、高信息密度、可信、实时。整体以中性深色工作台为主，青绿色代表可执行动作，蓝色表达信息，绿色只用于成功，琥珀和红色用于风险。禁止使用装饰性渐变、光球、玻璃拟态和大面积单一紫蓝色。

| `--text-secondary` | `#A8B1BA` | 次级说明 |
| `--text-muted` | `#737E88` | 时间、占位与弱信息 |

要求：

- 正文与背景对比度至少达到 WCAG AA 4.5:1；大字和非文本控件至少 3:1。
- 状态不能只靠颜色，必须同时具备图标或文本标签。
- 主色只用于可操作重点和活动状态，避免整页泛青。
- 后续 Light Theme 使用语义 Token 映射，不在组件内硬编码颜色。

### 8.3 字体、间距与形状

- UI 字体：`Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`。
- 代码字体：`JetBrains Mono, "SFMono-Regular", Consolas, monospace`。
- 字号建议：12（辅助）、13（紧凑表格）、14（正文）、16（区块标题）、20（页面标题）。不使用随视口缩放字号。
- 字重：400 正文、500 控件与次标题、600 页面标题；避免大面积粗体。
- 字间距统一为 0。
- 采用 4px 基础间距：4、8、12、16、24、32。
- 控件高度：紧凑 28px、默认 32px、主要输入 40px；固定高度控件内文本必须完整可见。
- 圆角：输入与按钮 6px，卡片/浮层 8px，标签 4px；不使用大药丸形文本按钮。

### 8.4 组件层级

- 页面区块使用分隔线和背景层级，不把每个区块做成悬浮卡片。
- 卡片仅用于审批、产物、重复列表项和需要明确边界的工具结果。
- 主按钮只在每个局部任务中保留一个；危险操作始终使用红色语义样式。
- 运行中状态使用小型 `LoaderCircle` 动效，不让加载动画改变布局尺寸。
- Tooltip 延迟 400-600ms；菜单、弹窗和抽屉支持 Escape 关闭及焦点归还。

## 11. 工程实施阶段

以下周期按 1 名产品/设计、2 名前端/全栈、1 名后端/平台、1 名 AI/集成工程师估算，可并行推进。

| 阶段 | 预计周期 | 主要产物 | 退出条件 |
| --- | --- | --- | --- |
| P0 需求与风险基线 | 2-3 天 | 产品蓝图、术语表、竞品差异、风险清单 | MVP 边界、三条核心闭环和角色达成一致 |
| P1 低保真与交互 | 3-5 天 | IA、页面流、线框、状态矩阵 | 四个演示场景无断点，关键状态齐全 |
| P2 高保真可点击原型 | 5-7 天 | 设计 Token、组件、图标映射、响应式原型 | 通过产品、设计和 3-5 位目标用户评审 |
| P3 工程底座 | 5 天 | 前端框架、路由、组件层、Mock API、质量门禁、ADR | Build/Lint/Test 通过，核心页面可独立开发 |
| P4 核心后端与 Agent 运行 | 3-4 周 | 领域模型、Session/Run、环境执行、流式事件、审批 | 人工任务可在测试仓库生成经验证的 Draft PR |
| P5 GitHub/Slack 闭环 | 2-3 周 | OAuth/App、Webhook、幂等、回写、深链、重试 | 两条外部触发链路端到端通过故障测试 |
| P6 安全、可靠性与 Beta | 1-2 周 | RBAC、Secret 管理、审计、监控、压测、运行手册 | Beta 验收标准通过，可供设计合作客户使用 |

### 11.1 推荐工程原则

- API 与事件契约优先：前后端共享 Session、Run、Step、Approval 状态枚举。
- UI 使用流式事件更新，但以服务端持久状态为准；断线重连后可恢复。
- 所有 Webhook 先落库、验签、去重，再异步处理；失败支持有上限重试和死信诊断。
- Agent 工具调用通过统一 Policy 层授权，不允许具体工具绕过审批策略。
- 外部写操作使用幂等键，创建 PR、评论和审批回写均可安全重试。
- 关键架构决策以 ADR 记录：执行隔离、事件模型、秘密管理、模型路由、日志保留。
- 模型供应商保持可替换；产品领域对象和 UI 不以某个模型名称建模。

### 11.2 测试分层

- 单元测试：状态转换、触发器匹配、权限策略、风险分级、幂等键。
- 组件测试：表单校验、时间线分组、审批状态、错误与空状态。
- 合约测试：GitHub/Slack Payload、API Schema、流式事件顺序。
- 集成测试：仓库克隆、工具执行、PR 创建、Webhook 重试、Secret 脱敏。
- 端到端测试：四个原型演示场景对应的真实测试环境流程。
- 安全测试：Webhook 验签、越权、Prompt Injection、路径逃逸、Secret 泄漏和日志脱敏。

## 12. 验收标准

### 12.1 原型验收

- 四个演示场景都能从入口走到明确结果，不存在无响应的主按钮或死链。
- 1440x900、1024x768、390x844 三个视口下无文本遮挡、控件溢出或布局跳动。
- 小屏可完成“查看会话、查看风险、批准/拒绝”，复杂配置明确引导到桌面端。
- 所有页面具备适用的加载、空、错误、权限不足和成功状态。
- 所有图标遵守 Lucide 映射、尺寸和可访问性规则；无 Emoji 充当功能图标。
- 键盘可完成导航、打开菜单、提交表单和处理审批；焦点清晰且顺序合理。
- 正文和关键控件达到 WCAG AA；状态同时使用文本/图标，不只使用颜色。
- 视觉色值来自 Token；没有散落的硬编码主题色、装饰性渐变或嵌套卡片。
- 原型文案使用一致术语，并能清楚回答“正在做什么、为什么暂停、下一步是什么”。

### 12.2 MVP 功能验收

- 新用户在 10 分钟内完成 GitHub 连接、选择模板并在测试仓库启动首次会话。
- 手动、Slack、GitHub 三类来源均能创建幂等 Session，并保留来源深链。
- Session 断线重连或页面刷新后状态一致，运行日志与产物可恢复查看。
- PR Author 能在隔离环境修改代码、运行指定检查并创建 Draft PR。
- Reviewer 能基于固定 Head SHA 输出带证据的 Review；SHA 变化会使旧审批失效。
- 高风险外部写操作未经有效审批不能执行，审批结果可在 Web 和 Slack 间同步。
- 所有外部写操作、审批和权限错误都可通过事件日志追踪到关联 Session 与操作者。
- Secret 在数据库、日志、错误、前端 Payload 和分析事件中均不出现明文。

### 12.3 工程质量验收

- 主分支持续通过格式化、Lint、类型检查、单元测试、关键 E2E 和生产构建。
- 领域状态转换由服务端校验，客户端不能通过构造请求跳过审批。
- Webhook 具备验签、去重、重试和死信诊断；重复事件不产生重复 PR/评论。
- 关键接口有结构化日志、指标和 Trace ID；可从 Session 追踪到工具和外部 API 调用。
- 核心页面交互反馈小于 100ms；流式事件到达后 1 秒内可见；长列表采用虚拟化或分页。
- 执行环境有 CPU、内存、时间、文件系统和网络边界，并可强制终止。
- 上线前完成威胁模型、最小权限审查、数据保留策略和事故响应手册。

| --- | --- | --- |
| Time to First Value | 注册到首个成功产物的时间 | 中位数小于 15 分钟 |
| Session 成功率 | 无人工接管且达到验收条件的会话占比 | 按专家与任务类型分层观察，不追求虚高总值 |
| PR 接受率 | Agent PR 被合并或仅小改后合并的比例 | 持续上升，同时监控回滚率 |
| 人工介入率 | 除策略审批外需要人工修复的会话比例 | 随模板成熟度下降 |
| 审批等待时长 | 从请求审批到有效决策的时长 | Slack 渠道显著短于仅 Web 通知 |
| 自动化复用率 | 同一专家被多成员、多次调用的频率 | 验证专家沉淀价值而非一次性 Demo |
| 安全事件率 | 越权、Secret 泄漏、错误外部写入 | 必须为零，近失事件也需要记录复盘 |

## 14. 原型后的决策门

进入真实 MVP 开发前必须用目标客户验证以下假设：

1. 用户是否把“专家”理解为可复用职责，而不是另一个聊天人格。
2. 时间线的信息层级是否足以建立信任，又不会因工具日志过多而失焦。
3. 团队更愿意在 Slack 处理哪些审批，哪些必须回到 Web 查看完整证据。
4. PR Author、Reviewer、Fixer 三专家串联是否比单一 Agent 更易理解和治理。
5. 哪些动作可默认自动执行，哪些必须由组织策略强制审批。

若上述问题没有形成稳定答案，不应提前投入拖拽编排器、插件市场或复杂计费系统。
